# Control Center v2 — Plan (v1.4, 2026-04-24)

**Principle:** v2 is a **clean standalone rewrite**, not a merge of v1's pieces.
v1 at `~/.config/control-center/` keeps running while v2 is built; once v2
passes its smoke test, v1 is archived to `~/.config/control-center.v1.archive/`
and v2 takes the same port. No long-term mixing. No "compatibility bridge."

## Why v2 exists

v1 works for two agents and one user but has accumulated:
- five directories, two settings files, ad-hoc env vars
- bash launcher with pidfiles + osascript
- single-file HTML dashboard
- in-memory `Map`s for state, no persistence guarantees
- bridges sharing a script and disambiguating by env var
- MCP-as-reply-channel that turned out to be unnecessary
- a local-LLM (qwen via spice-harvester) experiment that was too laggy to use

v2 fixes the architecture, not just the code.

## Goals

1. **One command** — `cc up` starts server, dashboard, all bridges, watchdog, pane-mirror, tmux sessions, terminal windows. `cc down` kills it all. `cc toggle` flips state.
2. **One config** — `~/.cc/config.toml` replaces scattered env + hardcoded paths. Env still overrides.
3. **One repo** — npm workspace monorepo. `npm i -g` works on a clean Mac.
4. **Tmux-only worker comms** — workers receive prompts via tmux inject, reply by typing in their pane. Pane-mirror forwards pane output to the channel. **No MCP from workers.**
5. **Bridge plugin interface** — any tmux-hosted CLI plugs in with a 30-line adapter. No REST-API "agents" — those are dead weight when the CLI is right there in tmux.
6. **Protocol as library** — `CLAUDE.md` / `GEMINI.md` generated from one template. Peer list auto-discovered from config.
7. **SQLite persistence** — survives pipeline growth past a few hundred tasks without memory pressure. Auto-migrates from v1's JSON state file once on first boot.
8. **Observability** — unified `GET /api/events` stream; per-task timeline in the dashboard; pane-mirror provides raw worker-pane transparency.

## Scope decisions (v1.4)

These are **explicit non-goals** for v2 — DO NOT add them back without rewriting this section:

- ❌ **No MCP server.** Workers don't need an outbound MCP tool — they type their reply in the tmux pane and pane-mirror posts it. Dashboard issues HTTP `/api/messages` + `/api/tasks` directly. (A read-only `cc` MCP for *Claude Desktop* chat-side tools could be added later as a separate package, but it is not part of the v2 control loop.)
- ❌ **No REST-API agents.** No `agents/gemini-rest`, no `agents/claude-api`. Workers are tmux panes hosting the official CLIs (`claude`, `gemini`). The bridge injects, the mirror reads. That's the entire agent layer.
- ❌ **No qwen / no local-LLM workers.** Spice-harvester / Ollama experiments stay outside v2. Too laggy to be a real worker. spice-harvester is disconnected from CC, not deleted (it has unrelated uses).
- ❌ **No aider.** What we needed aider for was reducing git-command hangs. The actual root cause was interactive prompts (`Username for...`, `Continue (y/n)?`) — solved by a small `git-helper` tool that runs git non-interactively (`GIT_TERMINAL_PROMPT=0`, `--no-pager`) with hard timeouts. ~80 lines, zero new dependencies.
- ❌ **No multi-user / clustering.**
- ❌ **No web auth.** Localhost shared-secret is enough.
- ❌ **No Rust/Go rewrite.** Node + SQLite + tmux.

## Future (post-v2)

- **Multi-gemini worker pool:** spawn N `gemini-cc-k` tmux sessions, each with its own bridge registered as `gemini-k`. CC dispatcher round-robins across idle workers (respects `capacity`). Quota-aware — pool shrinks on rate-limit errors.
- **Per-worker cost/quota telemetry:** track tokens/quota spent per task.
- **Hot-swap models:** `cc model @gemini gemini-2.5-pro` for one-off heavy work.
- **Read-only chat-side MCP:** `cc` MCP for Claude Desktop that exposes `cc_status`, `cc_read_messages`, `cc_dispatch_task`. Strictly outbound from chat → CC; never from CC → workers.

## Architecture

```
control-center-v2/
├── bin/cc                              # the ONE command (bash → packages/cli/src/cc.js)
├── packages/
│   ├── cli/                            # `cc up|down|toggle|status|logs|attach|send|doctor|init`
│   ├── server/                         # HTTP broker — routes/sse/tasks/persistence(SQLite)
│   ├── dashboard/                      # Alpine.js + Tailwind, single static bundle
│   ├── bridges/                        # tmux-bridge (the only bridge type for v2)
│   ├── agents/                         # watchdog + pane-mirror (NOT worker LLMs — those are tmux CLIs)
│   ├── shared/                         # CC client, SSE utils, protocol renderer, types
│   └── tools/                          # git-helper (non-interactive git wrapper called by workers)
└── workspaces/                         # rendered per-agent CLAUDE.md/GEMINI.md (generated by `cc init`)
```

**Key shape change vs. v1.0 plan:**
- `packages/agents/` no longer contains worker-LLM adapters. It contains the **infrastructure agents** (watchdog, pane-mirror) that observe and forward worker output.
- `packages/mcp/` deleted from the tree.
- `packages/tools/` added for `git-helper`.

## Protocol v1.4 (tightening of v1.3)

Changes from v1.3:
1. **Worker prompt template drops `cc_send_message`.** Replace with: "When you have something to say to the channel, type it in your pane like a normal CLI message. The mirror will publish it. Do not call MCP tools."
2. **Worker prompts must include explicit `cd <abs-path>` step.** Gemini-cli resolves paths relative to its workspace root, not absolute paths in instructions. Bit us on P3.1.
3. **Reviewer prompts declare the inline AC block as the single source of truth.** Reviewers do NOT consult WORK_PACKETS.md; they consult the AC block in the dispatch payload. This was the calibration miss on P3.1 cycle 1.
4. **Reviewer verdicts are binary.** "CONDITIONAL PASS" is banned. If there are required fixes, verdict = FAIL with a fix list.
5. **Worker ACK step preserved from v1.3.** Final action = `cc_update_task(id, status="completed", result="<1-line>")`. (This is HTTP, not MCP — task control is server-side, not chat-tool.)

## Work-packet contract

Every packet dispatched to @gemini follows the Universal Code Contract:

```json
{
  "files": [{ "path": "...", "content": "..." }],
  "deps": { "package": "^version" },
  "usage": "one-paragraph how-to",
  "notes": "trade-offs, TODOs, things I couldn't decide"
}
```

Packets are small enough to fit in one Flash response (~20KB). Large
packages (like `server/`) are split into sub-packets with explicit
integration points. **No packet exceeds ~250 lines of expected output.**

## Cutover from v1 to v2

When R12 smoke passes:
1. `ccctl down` (kills v1 stack)
2. `mv ~/.config/control-center ~/.config/control-center.v1.archive`
3. `cc init` (writes `~/.cc/config.toml`, renders protocol files)
4. `cc up` (starts v2 stack on the same port)
5. v2 server auto-imports `~/.control-center-state.json` once into SQLite
6. Dashboard URL is unchanged. Bridges/watchdog/mirror now run from v2 packages.

If anything fails: `cc down && mv ~/.config/control-center.v1.archive ~/.config/control-center && ccctl up`. v1 is one rename away the whole time.

## Success criteria for v2

1. On a clean Mac: `git clone && npm i && npm link && cc init && cc up` → dashboard at localhost:3000, all infrastructure agents online, claude+gemini tmux sessions live.
2. `cc send @gemini "ping"` → reply in channel within 3s (typed by gemini in its pane, picked up by mirror).
3. `cc doctor` → all green (node version, tmux installed, ports free, API keys present, config valid).
4. Kill the server mid-task → restart → task state survives, orphaned dispatched tasks re-dispatched on bridge reconnect with `?since=<id>`.
5. Dashboard feature parity with v1 minus the now-removed @qwen / MCP-call paths.
6. **No process is started outside `cc up`.** No "manual on the side." If a future component is added, it's registered with the CLI in the same change. (See `memory/factory_kill_switch.md`.)
